A Brief Introduction

Recently, I have gradually developed intensive interests in container related technology and tried to leverage Ansible and docker to solve the various SDLC issues faced by my current project. One of them is about how to gurantee consistencies across distinct environments so that the excuse of “This works on my machine” could be eliminated completely. Another one is about how to setup local dev environment effortlessly when newcomers join the team. Ideally, only two steps should be required before spinning up a local development environment: 1) git clone repo 2) run a script such as docker-composer up. Then, there you go!

After spent a while digging into Ansible and by trying my best to follow the recommended best practices, I managed to come up with various working ansible playbooks and roles to automate the current manual process of system provisioning. The following diagram depicts well about the renovated structure.

The Base Role

To start with, the base role serves as a jumping off point for all other roles, which provisions some common dependencies and configurations such as:

  1. vim configuration

  2. git configuration

  3. system performance monitor tools: iftop, htop, lsof

MOTD

One of the most awesome tasks is called motd.yml shown as below.

MOTD is in fact a unix custom and an acronym for “message of the day”, the purpose of which is to display dynamic data about the box from various aspects. For example, it could be the current system load, software upgrades notification, to name but a few.

Things could be made more humorous and instructive by adding the following message template to be displayed to whoever has just sshed to the box.

So, the final outcome is that the moment you have logged into a running box, a random wisdom quote, which is selected out of hundreds of millions, will be shown to you. As login counts increase, one could bet that there would definitely exist some quotes that you will love, read it out aloud and share with your friends.

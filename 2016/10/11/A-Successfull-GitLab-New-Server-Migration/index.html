A-Successfull-GitLab-New-Server-Migration

Posted on 2016-10-11   |   In gitlab , linux administration   |  

In this blog, I will show you how to migrate and upgrade an existing GitLab instance to a different VPS provider from a real practice with tips and lessons learned.

Problems

A graph is worth than 10K words, right?!

Goals

To tackle those aforementioned problems, the following goals are in our top priorities:

  1. Speeding up developers’ GitLab access by relocating the GitLab server to a nearer geographical location. As of now, we have no choice but to choose the mediocre AliYun as our cloud provider for the testing environment. Hopefully, AWS could show its presence in the mainland china asap.

  2. Reducing the time taken for Canary environment deployment, which is intensively used by the QA fellas.

How

  1. Installing GitLab software with the identical version 8.3.2-ce.0 in a newly purchased server located in AliYun HZ.

  2. Performing a backup operation in the old running GitLab instance. The backup task will dump the data of all the existing databases, repositories, uploads, builds, artifacts, lsf objects.

  3. Transferring the backup tar file from the old server to the new server.

  4. Restoring the transferred backup in the new server.

  5. In order to allow existing users to access the new server, we need to transfer serval configuration files as well.

  6. Updating the new gitlab instance to the latest version.

  7. Flipping the gitlab domain to the newly equipped server address.

Caveats

Continuous Network File Transfer

Per my experience, the above step 3 took the longest time since the network connection between the two servers is not that fast due to the bandwidth limit imposed on the Ali Yun Box. The average transfer speed hit around 90K/s and it cost more than one hour to complete the transfer of the backup tar file with the size of 763MB. So what if network goes down at the end of the transfer? To start it all over again? No way! SSH Client - Known Host Change

After the domain has changed its target ip address, ssh clients will be effected. This is due to the fact that the ~/.ssh/known_hosts file contains an stale entry regarding the domain since the fingerprint has changed, thus ssh client complains about it. There are two approaches to fix it and retry afterwards: either 1) manually delete the corresponding entry from the file or 2) run the command for automatic removal of the entry ssh-keygen -f “/home/gitlab-runner/.ssh/known_hosts” -R {domain}
